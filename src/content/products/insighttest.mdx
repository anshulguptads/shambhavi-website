## AI capabilities — four things our AI does that hand-coded QA doesn't

### 01 · AI Test Generation — describe the behavior. Get the test.

Feed a user story, a Figma flow, a Jira ticket, or an existing screen recording — our NLP engine generates the test cases, selects appropriate assertions, and populates realistic test data. What used to be a day of scripting is now a 10-minute review.

```
input →  "User should be able to apply a promo code and see the updated total at checkout."
output → 4 test cases generated · 12 assertions · 3 edge cases identified
```

### 02 · Self-Healing Automation — the test adapts when your UI doesn't

When a developer renames a class or restructures a component, traditional test suites turn red. InsightTest AI's healer analyzes the DOM, identifies the new element using visual, semantic, and positional signals, and updates the selector automatically. The test keeps running. The engineer gets a notification, not a 2 AM page.

```
healed → .checkout-btn (not found) → #pay-now (visual + semantic match, 96% confidence)
```

### 03 · Anomaly Detection — the real regression, not the flaky noise

Not every red test is a bug. InsightTest learns your suite's baseline — timing, failure patterns, flake rate per test — and distinguishes genuine regressions from environmental flukes. Your team triages real issues, not yesterday's network hiccup.

```
flagged → Login test · 12% latency spike vs 30-day baseline · likely regression (not flake)
```

### 04 · Analytics Validation — events fire where they should, or you know immediately

Marketing teams depend on event tracking nobody validates until the campaign is already live and broken. InsightTest runs scheduled validation on your Adobe, RudderStack, and MoEngage pipelines — firing events, checking payloads, confirming downstream receipt. Marketing sleeps easier.

```
verified → add_to_cart · RudderStack · payload intact · 3 downstream destinations OK
```

## Architecture — built for serious engineering, priced for rational teams

InsightTest AI is engineered on a **serverless-first microservices architecture** designed to scale from a 5-person team to a 500-engineer organization without architectural rewrites.

The execution layer runs on **AWS Lambda**, spinning up test workers on demand — you pay for executions, not idle capacity. The AI layer combines NLP models for test-intent parsing and purpose-built anomaly detection models trained on QA-specific signals.

**Deploy anywhere.** The full platform runs on AWS, Azure, or lightweight Railway for smaller teams — or on your own VPC for enterprises that require it.

## FAQ — questions engineering leads ask us

### Is the 60-70% effort reduction claim real?

It's measured across teams using InsightTest for three sprints or more, comparing time spent writing and maintaining tests against their prior baseline. The biggest gains are from self-healing (which eliminates most maintenance cycles) and AI generation (which removes the from-scratch writing cost). Smaller suites see smaller gains; mature suites with hundreds of tests see the largest.

### Can non-engineers actually write useful tests with this?

Product managers, designers, and QA analysts can build meaningful end-to-end tests through the drag-and-drop builder and natural-language generation. For complex assertions, data-driven scenarios, and deep API contract testing, engineers still drive. The point isn't to replace engineers — it's to let every team member contribute coverage they couldn't before.

### How does self-healing know it hasn't "healed" to the wrong element?

Every auto-healed selector gets a confidence score from visual, semantic, and positional analysis. Below a configurable threshold (default 85%), the test pauses for human review rather than proceeding silently. Every heal is logged, reversible, and auditable — so if the platform ever makes a bad call, you see it before it lands in your report.

### Will this replace our existing test framework (Playwright, Cypress, Appium)?

It doesn't have to. InsightTest can coexist with your existing frameworks — some teams use it for new feature coverage while keeping their legacy suites running, others migrate gradually. We support import from Playwright, Cypress, Selenium, and Appium test definitions so migration is incremental, not a big bang.

### Where does our test data go? Is it secure?

Test data is tenant-isolated and encrypted at rest and in transit. For regulated industries, we offer self-hosted and private VPC deployments so test data never leaves your infrastructure. The AI models that parse test intent never see production customer data — only test artifacts.

### How does pricing work? Per user, per test, per execution?

Our pricing combines a platform fee based on team size (unlimited test authoring) with a pay-per-execution model for test runs (tiered by volume). No surprise overages — budget caps and alerts are built in. Enterprise plans offer fixed-cost execution pools. Contact our team for a quote specific to your scale.
